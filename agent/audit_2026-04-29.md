# Audit Report — Diaphragmatic Endometriosis Training Stack (2026-04-29)

Scope: training + evaluation stack for the 2.5D detector (no EDA). This audit emphasizes software engineering efficiency (GPU starvation risk, throughput, reuse) and research integrity (leakage, metric validity). No code changes were made.

Sources reviewed: `agent/complete_spec/00_PRD.md`, `agent/complete_spec/03_dataset_datamodule.md`, `agent/complete_spec/04_augmentation.md`, `agent/complete_spec/05_sampler_hnm.md`, `agent/complete_spec/06_model_training.md`, `agent/complete_spec/06_5_gru_rescorer.md`, `agent/complete_spec/07_post_training_eval.md`, `agent/complete_spec/IMPLEMENTATION_LOG.md`, recent git history, and runtime code under `endo/`.

**Executive Summary**
- The stack is broadly aligned to spec and trains end‑to‑end, but there are clear GPU‑starvation risks from heavy CPU augmentation and per‑batch NMS in training. These can reduce throughput on A10/L40S unless you raise workers or reduce CPU load.
- Evaluation is vulnerable to optimistic bias: WBF thresholds are tuned and evaluated on the same validation data, and FROC uses a proxy GT volume for positives unless GT masks are explicitly passed. These issues can inflate reported sensitivity/AUROC/AP relative to a strict review.
- Deep‑eval caches are not validated against the expected val pid set; a corrupted or mis‑produced cache can silently leak train/holdout pids into CV metrics.
- EMA usage is inconsistent: `PeriodicDeepEvalCallback` attempts a swap but `EmaCallback` exposes no swap API, so deep‑eval runs on live weights, while val uses EMA. This is a discrepancy for the cached eval source.
- Several guardrails remain manual (bf16 stability, holdout re‑touches, assigner parity), so results are credible but not “publication‑grade safe” without a few minimal integrity fixes.

**Engineering Efficiency Findings (GPU starvation & reuse)**
- Heavy CPU augmentation likely dominates batch prep: `scipy.ndimage.affine_transform` + per‑slice `map_coordinates` + per‑paste `distance_transform_edt` (`endo/augmentation/geometric.py:122`, `endo/augmentation/geometric.py:220`, `endo/augmentation/paste.py:238`). Expect GPU idle time unless `num_workers` is high and `prefetch_factor` is tuned. Recommendation: profile dataloader throughput and consider reducing elastic/paste frequency or gating on a lower probability if throughput is poor.
- Per‑batch NMS during training adds nontrivial GPU work: `LesionDetectorLM._update_score_ema` calls `head.predict` every training step (`endo/lightning_module.py:123`). For throughput, update EMA every N steps or use an aux‑seg proxy for hard‑negative mining if acceptable.
- Double copy of full volume for augmentation: dataset casts to float32 contiguous then augmentation re‑casts (`endo/data/dataset.py:251`, `endo/augmentation/transform.py:262`). This doubles memory bandwidth cost per sample. Recommendation: cast only once (either in dataset or in augmentation) to reduce CPU load.
- Large per‑sample memory footprint: `volume_full_cropped` is float32 `(384,160,384)` (~94 MB) per training sample (`endo/data/dataset.py:251`). This makes CPU RAM and bandwidth the bottleneck. If acceptable, keep fp16 until the last possible point and only upcast when needed.
- Dataset RNG is per‑dataset instance and not worker‑seeded (`endo/data/dataset.py:121`), so multi‑worker dataloading can repeat identical jitter sequences across workers. Recommendation: seed per worker or derive jitter from `(patient_id, slice_y, epoch)`.
- `inference_pass` ignores its `batch_size` argument, always using the DataModule batch size (`endo/inference_pass.py:44`). This is confusing and prevents controlling inference throughput from callers; either wire it through or remove the parameter.
- `cohort_path` in `LesionDataModule` is stored but never used (`endo/data/datamodule.py:61`, `endo/data/datamodule.py:116`). If `cohort.json` is authoritative, holdout membership may diverge from `manifest.jsonl` without any warning.
- Minor collate overhead: `np.stack(...).float()` creates extra copies per batch (`endo/data/collate.py:19`). Small, but adds up.
- Spec precheck is missing: the preflight checks in `agent/complete_spec/06_model_training.md` (cache integrity, assigner parity, RAM/GPU bounds) are not implemented. This increases failure risk and makes troubleshooting slower.

**Research Integrity Findings (leakage & metric validity)**
- Threshold tuning uses the same data for evaluation, inflating metrics. Per‑fold and pooled WBF thresholds are chosen on the val set and evaluated on that same data (`endo/eval/run_eval.py:269`, `endo/eval/run_eval.py:322`, `endo/eval/threshold_search.py:38`). Recommendation: cross‑fold thresholding or nested CV; at minimum report “raw (no threshold tuning)” metrics alongside “tuned” metrics.
- CV‑pooled thresholds are optimized and evaluated on the same pooled val set (`endo/eval/run_eval.py:322`), further inflating cv_pooled results. Recommendation: leave‑one‑fold‑out thresholding, then average.
- Deep‑eval caches are not validated against expected val pids (`endo/eval/run_eval.py:76`, `endo/sampler/periodic_eval.py:224`). A wrong cache can silently introduce train/holdout pids into evaluation. Recommendation: store val pid list in the npz and verify on load; hard‑fail on pid mismatch.
- FROC uses a central‑cuboid proxy GT when no lesion masks are provided (`endo/eval/froc.py:61`). This is not lesion‑level detection FROC and can overstate detection performance. Recommendation: use GT lesion masks/boxes from cache for true lesion‑level FROC or label this metric explicitly as a proxy.
- AUROC/AP are volume‑level classification metrics computed from per‑volume max score, not detection‑level AP (`endo/eval/metrics.py:159`). This is valid, but must be reported clearly as volume‑level AUROC/AP (not detection AP).
- FROC CIs use a per‑volume score bootstrap rather than the detection map used for FROC point estimates (`endo/eval/metrics.py:209`, `endo/eval/froc.py:128`). This mismatch can understate/overstate CI width. Recommendation: either bootstrap on the same detection map or label CIs as approximate.
- Holdout can be touched repeatedly; no guard file blocks repeated runs (`endo/eval/run_eval.py:434`). This risks implicit test‑set tuning. Recommendation: add a guard file in `runs/<exp>/holdout/` and require an explicit override flag for re‑touching.
- EMA inconsistency in deep‑eval: `PeriodicDeepEvalCallback` tries to swap to EMA but `EmaCallback` exposes no swap API (`endo/sampler/periodic_eval.py:175`, `endo/ema_callback.py:72`). Deep‑eval runs on live weights while val uses EMA; the cached eval source differs from the ckpt that may later be used. Recommendation: expose an EMA swap API or move deep‑eval to run inside the EMA swap window.
- Latest deep‑eval cache is used, not best epoch (`endo/eval/run_eval.py:68`). If the latest epoch is worse than best, CV metrics may under‑estimate. Recommendation: select deep‑eval cache matching the best checkpoint epoch, or re‑infer with best ckpt before final CV report.

**Additional Observations (spec deviations & minor issues)**
- `ScoreEMATracker` ignores `experiment.sampler.score_ema_decay` (config value unused); created with default 0.9 in CLI (`endo/cli/run_experiment.py:236`). If you want reproducible HNM behavior across configs, pass the configured decay.
- Augmentation seeding is deterministic per `(patient_id, slice_y)` (`endo/augmentation/transform.py:182`), meaning the same slice sees the same augmentation every epoch. This is spec‑compliant but may reduce augmentation diversity; consider including epoch in the seed if you want more variety.

**Recommended Minimal‑Change Action Plan (ordered)**
1. Add deep‑eval pid validation on load and/or embed expected val pid list in the npz; hard‑fail on mismatch. (`endo/eval/run_eval.py`, `endo/sampler/periodic_eval.py`)
2. Decouple threshold tuning from evaluation (cross‑fold thresholds or raw‑metric reporting) and label “tuned” metrics clearly. (`endo/eval/run_eval.py`, `endo/eval/threshold_search.py`)
3. Provide an EMA swap API and ensure deep‑eval uses EMA weights, or document that deep‑eval is on live weights. (`endo/ema_callback.py`, `endo/sampler/periodic_eval.py`)
4. Fix dataset RNG seeding for multi‑worker jitter diversity. (`endo/data/dataset.py`)
5. Reduce augmentation copy overhead and/or gate heavy transforms; profile dataloader throughput to confirm GPU utilization. (`endo/data/dataset.py`, `endo/augmentation/transform.py`, `endo/augmentation/geometric.py`, `endo/augmentation/paste.py`)

**Suggested Metrics to Add for Scientific Reporting**
- Lesion‑level sensitivity and FP/vol using true lesion masks/boxes, not proxy GT (if available from cache).
- Stratified metrics by lesion size (e.g., <5 mm, 5–10 mm, >10 mm) to demonstrate clinical relevance for tiny plaques.
- Calibration curves or Brier score for volume‑level scores (useful for clinical decision support claims).
